HELIX 当前代码实现说明

当前仓库已经具备一个可运行的 plan-only agent 主流程，并预留了图构建、ToolCall、 GraphPatch、MemoryHealthCompiler、Controlled Recall、HookBus 和自进化等核心边界。

项目：D:\workspace\HELIX 架构源：HELIX_architecture_v7.md 当前模式：plan_only 测试：72 passed

现在有几个 Agent？

代码层面：一个 HELIX agent 系统

当前代码里没有新增架构外 agent。存在的是多个内部 workflow 和模块边界， 它们共同组成 HELIX agent 系统。

Workflow 不等于新 Agent

`graph-construction bootstrap workflow`、`capability evolution`、`memory evolution` 都是内部流程，不是额外的独立 agent。

顶层模块结构

appCLI 入口，当前提供 `helix plan`。
config外置配置和 prompt 配置。
schemas跨模块共享的 Pydantic 对象契约。
core核心服务，目前包含保守 TaskFingerprinter。

hooksHookBus 生命周期事件总线。
graphL0/L1 存储协议和图层策略。
graph_patchGraphPatch 构建、校验和审计。
graph_healthMemoryHealthCompiler 和生命周期管理。
graph_construction既有图构建 workflow 的 bootstrap 模式。
capability_evolution候选工具 / workflow 和 ToolCallSpec 构建边界。
memory失败事件到候选 constraint 的经验沉淀。

图构建 Bootstrap 流程

这不是新 agent，而是 HELIX 既有图构建 workflow 的 bootstrap 运行模式。 等真实数据库 adapter 接入后，第一次完整构建 L0/L1 会走这条受控路径。 

flowchart TD
    A["GraphPatch candidate"] --> B["GraphPatchAuditor"]
    B --> C{"audit pass?"}
    C -- "no" --> D["blocked before L0 write"]
    C -- "yes" --> E["FullGraphStore.apply_patch"]
    E --> F["MemoryHealthCompiler"]
    F --> G["L1 materialization report"]

实用总结

当前 HELIX 可以做什么

它可以跑通受控 plan-only 主链路，拒绝编造工具和 workflow，并把运行过程记录成结构化事件。

当前 HELIX 还不能做什么

它还不能执行真实生信工具、查询真实图数据库、构建真实 L1 healthy graph，或基于真实证据输出领域 workflow。
